## 1. 概要

仕事中などチャートを継続して見られない時間でも、システムが株価をリアルタイムに監視し、条件成立時に**自動で売買判断・発注・決済**を行う、日本株向けの**自動デイトレード・エンジン**である。

単なる指値・逆指値の置き注文ではなく、**場中に複数回トレードする**ことを目的とする。売買判断の主体は Python による数値監視・条件判定であり、AIは「シグナル発生時の補助判断」「日次ログ分析」「改善提案」に限定して用いる。

### ゴール
- 取引時間中、対象1〜5銘柄を無停止で監視し、ルールに基づくシグナル検知・（仮想）発注・決済・損益記録が自動で回る。
- すべての判断・注文・エラーがDBとログに残り、後から検証できる。
- リスク上限（日次損失・取引回数など）に達したら自動で取引を停止する。
- 月額コストを抑える（目安：合計 5,000円前後〜1万円以内、MVPはこれを大きく下回る）。

### 目指さないもの
AIが相場の雰囲気で売買を全委託される「ギャンブル的ボット」ではなく、**ルールベースの自動売買エンジン ＋ AIによる補助判断・分析**を目指す（詳細は §2.2）。

## 2. 設計思想（基本方針）

### 2.1 ルールベース主体 ＋ AI補助
| 役割 | 担当 |
|------|------|
| 毎秒の株価監視・条件判定・発注・損切り・利確 | **Python** |
| シグナル発生時の補助判断・日次ログ分析・改善提案 | **AI**（Claude API 等） |

- **常時の監視はPython**が行う（高速・低コスト）。AIは毎秒・全銘柄に呼ばない。
- **AIが登場するのは「シグナルが出たその瞬間」だけ**。これによりコストとレイテンシを抑える。
- AIは助言（Go／見送り／サイズ調整）までで、**発注・損切りの最終権限は持たない**。ハードなリスク上限はAIでも上書きできない。

### 2.2 AIの関与レベルと採用方針

AIの関与は0か100ではなく段階で設計する。本システムは **Level 2.5（パターンB）** を採用する。

| Lv | AIの役割 | 発注・損切りの権限 | 採否 |
|----|---------|------------------|------|
| 0 | 使わない（日次ログ分析のみ） | ルール | コスト縮退時のフォールバック |
| 1 | シグナル時に Go/NO を助言 | ルール | （2.5の下位） |
| **2.5** | **ルールが候補を出し、その瞬間だけAIが数値を見て Go／見送り／サイズ縮小を判断。発注・損切りはルールが固定で執行** | **ルール（固定）** | **★採用（本システム）** |
| 3 | AIがエントリー候補そのものも提案。ルール／リスクが filter して執行 | ルール＋リスク | 将来オプション（要・効果測定） |
| 4 | AIが雰囲気で全部決定・全委託（檻なし） | AI | **対象外** |

- **採用（Level 2.5）の動作**：
  1. Pythonが毎秒監視し、ルール条件成立で**シグナル（買い候補）**を出す。
  2. **その瞬間だけ**、現在の数値をAIに渡し、**Go／見送り／サイズ縮小**を理由つきで判断させる。
  3. リスク管理ガード（固定ルール）を通し、**Pythonが買い注文と損切り注文を発注・管理**する。
- **Level 4を対象外とする理由**：損切り・撤退まで非決定的なAIに全委託すると、バグ・誤読時に損失を止める“檻”が外れ、暴走・誤発注のリスクが過大になるため。
- 将来、効果を測定したうえで **Level 3（AIが候補も提案）** へ拡張する余地を残す。

## 4. システム構成

### 4.1 構成図

エンジン本体（Python）は**OS非依存**。証券・データへの接続だけを差し替え可能な**アダプタ**で分離し、Mac/Windowsどちらでも動かす。

```
Mac（常時稼働）／ Python エンジン（OS非依存）

 [毎秒・全銘柄を監視] 市場データ監視（VWAP・出来高・移動平均・直近高値）
        │  条件成立（シグナル＝買い候補）した“その瞬間だけ”
        ▼
 [AI補助判断] 数値を渡し Go / 見送り / サイズ縮小（理由つき・タイムアウトあり）
        │  ※助言のみ。発注・損切りの権限は持たない
        ▼
 [リスク管理ガード] 日次/銘柄別の損失上限・取引回数・二重発注防止（固定ルール・AI上書き不可）
        │
        ▼
 [発注・約定管理 OMS] 買い注文 ＋ 損切り注文を固定ルールで発注・管理
        │
        ├─► 永続化（SQLite）   ：全シグナル・注文・約定・損益・AI判断
        ├─► 通知（Discord）    ：シグナル/約定/損切り/利確/エラー/日次結果
        └─► ポジション・口座照合：システム内部状態と証券口座の実態を定期照合

 接続は差し替え可能なアダプタ経由：
   DataFeedアダプタ … Step1-3 = 無料/遅延データ(yfinance等) ／ 本番 = リアルタイム
   Brokerアダプタ   … Step1-3 = PAPER(仮想) ／ 本番(Step4) = kabu(Windows経由) or IBKR(Mac完結)

 AI補助は「シグナル時のみ」→ AI API（Claude 等）
```

### 4.2 モジュール一覧
| # | モジュール | 役割 |
|---|-----------|------|
| 1 | 市場データ監視 | 価格・出来高・板の取得、指標計算（VWAP/MA等）。`DataFeedAdapter`経由 |
| 2 | シグナル判定（戦略） | ルールに基づくエントリー/エグジット条件の評価 |
| 3 | AI補助判断 | シグナル時に数値を渡し、Go/見送り/サイズを取得（タイムアウト付き） |
| 4 | リスク管理ガード | 全注文の最終関門。上限超過・異常を検知し発注ブロック/停止 |
| 5 | 発注・約定管理（OMS） | 発注・取消・約定確認・注文状態管理（冪等性確保）。`BrokerAdapter`経由 |
| 6 | ポジション/口座照合 | 内部状態と証券口座の実態を定期照合（リコンシリエーション） |
| 7 | 永続化 | SQLiteへ全データ保存 |
| 8 | 通知 | Discordへイベント通知 |
| 9 | スケジューラ | 寄付・前場・後場・大引け・強制決済の時間管理 |
| 10 | 制御 | 起動/停止/手動キルスイッチ、設定管理 |

### 4.3 アダプタ層（クロスプラットフォーム）
- 市場データ(1)とOMS(5)は `DataFeedAdapter` / `BrokerAdapter` インターフェース越しに実装し、`PAPER`（仮想）・`kabu`・`IBKR` などを差し替え可能にする。
- これによりエンジン本体は**OS・証券会社に非依存**となり、開発〜仮想売買はMac単体で完結する。

## 5. 機能要件

### 5.1 市場データ監視
- `DataFeedAdapter` 経由で対象銘柄の現値・出来高（板）を取得。実装を差し替え可能にする。
  - **Step1〜3（Mac）**：無料/遅延データで開発・検証（例：`yfinance`＝約15〜20分遅延、ティッカーは `6920.T` 形式／J-Quantsはヒストリカル＝バックテスト用）。遅延データはロジック構築用で、エントリー精度の検証はLIVEデータで行う。
  - **Step4（LIVE）**：採用ブローカーのリアルタイム配信（kabuの WebSocket PUSH、またはIBKRのリアルタイム）。
- 指標計算：VWAP、移動平均（例：5/25分）、当日高値・安値、直近高値、出来高急増判定。
- 受信断・データ欠損・異常値を検知（§8）。

### 5.2 取引戦略：VWAP順張り（MVP初期戦略）
- 戦略はプラグイン的に追加可能な構造（`strategy_id` で識別）。
- 初期戦略のルール（数値はバックテスト／仮想売買で調整するスタート値）：

| 区分 | ルール（初期パラメータ） |
|------|------------------------|
| エントリー | 価格がVWAPを**上抜け** ＋ 直近の**出来高増加** ＋ 当日**直近高値を更新**。寄り直後の板寄せ時間帯は見送り |
| 利確 | エントリー価格 **+0.5〜1.0%** 到達、または **VWAP割れ**で利を伸ばさず撤退 |
| 損切り | エントリー価格 **−0.5%**（固定）。逆指値で必ず置く |
| 時間切れ | 保有 **30分** で手仕舞い |
| 当日制約 | **同一銘柄は1日1往復**（現物・差金決済規制対応、§9.1）。クローズ後は当日再エントリーしない |
| 強制決済 | 大引け前（§5.10）に未決済を成行で手仕舞い |

- しきい値・%・分・MA期間などは**設定ファイルで外出し**する。
- まずは**買い（ロング）のみ**。空売り・他戦略（オープニングレンジ・ブレイク等）を後から追加できる構造とする。

### 5.3 AI補助判断（Level 2.5）
- **呼び出しタイミング**：ルールがシグナルを出した瞬間のみ（毎秒監視はPython）。
- **入力**：数値のみ（現値、VWAP乖離、出来高、直近値動き、当日損益、保有状況など）。AIに渡すためのチャート画像生成は行わない。
- **出力**：構造化（例 `{judgement: GO|SKIP|REDUCE, reason: str, confidence: 0-1}`）。
- **AI判断の性質（期待値の正しい設定）**：AIは「渡した数値 ＋ 一般的な市場知識」で推論する。**特定銘柄の過去を学習済みのオラクル（未来予知）ではない**。役割は「スプレッドが広い／板が薄い／荒れやすい時間帯」等を踏まえた**常識的な確認・見送りの横やり**である。過信しない。
- **必須制御**：
  - **タイムアウト**（例：2〜3秒）。期限内に応答が無ければ**保守的に「見送り」へフォールバック**。
  - AIの応答は通信＋推論で**“秒”の単位**かかる。戦略の時間軸はこれに耐える設計（極端に速いスキャルピングはAI判断と相性が悪い）。
  - AIの `GO` でも、リスク管理ガード（§5.4）の**ハード上限は上書き不可**。発注・損切りはPythonが固定ルールで出す。
  - 呼び出し回数・トークン・コストを記録（コスト監視）。
- **検証方針**：ルール部分は過去データでバックテスト可能。AIの判断部分は非決定的なため、**仮想売買（フォワードテスト）を長めに回し、全AI判断をログして当否を測る**。
- **モデル選定**：シグナル時の補助＝**低コストモデル（例：Claude Haiku）**／日次ログ分析・改善提案＝必要時のみ上位モデル（Sonnet/Opus）をバッチ実行。

### 5.4 リスク管理ガード（最終関門）
- すべての発注は本モジュールを通す。1つでも違反すれば発注をブロック、必要に応じ全体停止。
- チェック項目は §8 に詳述。

### 5.5 発注・約定管理（OMS）
- 機能：買い注文、売り注文、注文取消、約定確認、注文状態（未約定/一部約定/全約定/取消/失効）管理。
- **冪等性**：同一シグナルからの二重発注を防止（発注キー管理＋発注前に既存注文・ポジション確認）。
- エントリー約定と同時に**損切りの逆指値**を必ず発注。
- 未約定注文は一定時間で自動取消。

### 5.6 ポジション・口座照合（リコンシリエーション）
- 一定間隔（数十秒〜数分）で、APIから取得した実保有・残高・注文状態と内部状態を照合。
- 乖離検知時：通知＋（重大なら）新規発注停止。「持っていないと思っていたら実は持っていた」事故を防ぐ。

### 5.7 永続化（DB）
- SQLiteに全データ保存（テーブル設計は §7）。

### 5.8 通知
- **Discord Webhook**（無料・実装容易）。
- 通知イベント：シグナル発生／注文発注／約定／損切り／利確／エラー／取引停止／1日の結果サマリ。
- 重要度でレベル分け（INFO/WARN/CRITICAL）。

### 5.9 制御・キルスイッチ
- 起動／正常停止、**手動キルスイッチ（即時の新規発注停止）**。
- 設定（対象銘柄、戦略パラメータ、リスク上限、稼働モード）を外部ファイルで管理。
- 稼働モード：`PAPER`（仮想）／`LIVE`（実売買）を明示的に切替。**初期値は `PAPER`**。

### 5.10 スケジューラ（場の時間管理）
- 東証の取引時間（2024/11/5の延長後）を前提に状態遷移する。

| 区分 | 時間 | システム動作 |
|------|------|-------------|
| 寄付前 | 〜9:00 | 監視準備・板寄せ注意（新規エントリーは原則見送り） |
| 前場 | 9:00–11:30 | 通常監視・売買 |
| 昼休み | 11:30–12:30 | 監視一時停止・状態維持 |
| 後場 | 12:30–15:30 | 通常監視・売買 |
| 大引け前 | 例：15:20–15:25 | **保有玉の強制手仕舞い開始**（デイトレは持ち越さない） |
| クロージング・オークション | 15:25–15:30 | 引け成行等での手仕舞い完了を確認 |
| 引け後 | 15:30〜 | 約定確定・日次集計・AI日次分析・通知 |

> 強制決済の具体時刻は、スリッページ・流動性を見て調整する。

## 6. 外部サービス・技術選定

### 6.1 証券接続：`BrokerAdapter`
エンジンは証券会社に直接依存せず、`BrokerAdapter`（発注/取消/約定確認/建玉・残高取得）越しに接続する。

| アダプタ実装 | フェーズ | 稼働環境 | 備考 |
|-------------|---------|---------|------|
| `PaperBroker`（仮想） | Step1〜3 | **Mac単体** | 実発注せず約定をシミュレート。本番証券は不要 |
| `KabuBroker`（本命） | Step4〜 | Windows | 三菱UFJ eスマート証券のkabuステーションAPI。国内で実質唯一の自動売買API |
| `IbkrBroker`（代替） | 任意 | Mac完結 | Mac完結にこだわる場合のみ。外国証券で確定申告が必要 |

本番は **kabuステーションAPI（三菱UFJ eスマート証券＝旧auカブコム証券）** を採用する。日本株をプログラムから自動売買できる国内リテール証券は実質これのみ（松井・SBI・マネックス等は個人向けの株自動売買APIを提供せず、楽天はExcel用RSSのみ）。Step1〜3はMacで `PaperBroker` を用い、本番接続はStep4で行う。

**本命：kabuステーションAPI（三菱UFJ eスマート証券）**
- リアルタイム株価／銘柄監視／売買・取消／約定・建玉確認に対応。REST ＋ WebSocket(PUSH)。本番/検証でポートが分かれる（例：本番18080／検証18081）。検証環境（デモ）あり。
- 利用料：kabuステーション「Professionalプラン」で**無料**。条件は「**信用口座の開設＋3か月に1回の取引**」（初回は約2か月無条件）。※信用口座は開設するが、**取引は現物でよい**（§9.1）。
- レート制限：発注5件/秒、情報・余力・銘柄登録10件/秒（本システムには十分）。認証はAPIパスワード→トークン方式。PUSH登録は約50銘柄上限。
- **Windows必須**：APIは「kabuステーション本体と同一PC・同一IP」からのみ受け付ける。よって**kabuステーションとエンジンは同じWindows上で動かす**のが確実。

**実行環境：開発＝Mac／運用＝Windowsデスクトップ（確定）**
- **開発（Step1〜3）はMac（M1）**：ロジック実装・指標計算・`PaperBroker`での仮想売買（無料/遅延データ）。費用ゼロ。
- **運用（kabu接続テスト〜本番 Step4）は自宅のWindowsデスクトップPC**：kabuステーション本体とエンジンを**同じWindows上で稼働**（「同一PC・同一IP」要件を自然に満たす）。VM・ミニPC・VPSは不要。
- **コード同期**：Mac↔Windowsはgitリポジトリで同期（同一のPythonコードを両環境で動かす＝OS非依存設計、§6.5）。
- **kabu接続の検証**：本番前に、Windows上でkabuの**検証環境（デモ）**に接続して動作確認してから少額ライブへ。

**代替：Interactive Brokers（IBKR）** … Mac完結にこだわる場合のみ。IB GatewayがMacで動作し`ib_insync`等で自動売買可。ただし外国証券で特定口座なし＝確定申告が必要、市場データ料あり。

> 認証情報（ログインPW、APIパスワード、トークン等）はソースに直書きせず、環境変数や設定ファイル（gitignore対象）で管理する。

### 6.2 データベース：SQLite
- **SQLiteはDBサーバーではなく「エンジンと同じPCに置くローカルファイル」**（例：`trades.db`）。別途DBサーバや「DB専用マシン」は不要。
- したがってDBは**エンジンが動くPCに同居**する：開発時はMac上、**運用時はWindows上**。**運用中にMacを起動しておく必要はない**（kabu＋エンジン＋DBがすべてWindows内で完結）。
- DBファイルはgit同期の対象外（`.gitignore`）。**コードはgitで共有、データは各PCローカル**（開発用と本番用のデータが自然に分離）。
- 本番状況はDiscord通知で受け取れるため、Mac等からDBに接続する必要はない。将来、遠隔ダッシュボードが必要になれば Supabase（無料枠）/ PostgreSQL 等へ移行。

### 6.3 通知：Discord
- Discord Webhook：無料・URLにPOSTするだけ・実装容易。
- 代替：Slack Incoming Webhook、メール（SMTP）、LINE Messaging API（LINE Notifyは終了済みのため使う場合はMessaging API）。

### 6.4 AI：Claude API
- 補助判断（Level 2.5）は低コストモデル（例：Claude Haiku）。日次分析・改善提案は必要時のみ上位モデルをバッチ実行。
- **構造化出力**で受け取り、ルールエンジンが機械的に扱えるようにする。
- コスト・レイテンシ・失敗時フォールバックを必ず計測・実装。

### 6.5 言語・主要ライブラリ
- 言語：Python 3.x（Mac/Windows両対応のOS非依存コード。パス・改行・プロセス管理に注意）。
- 想定ライブラリ：`httpx`/`requests`（REST）、`websockets`（PUSH）、`pandas`/`numpy`（指標計算）、`pydantic`（設定・スキーマ検証）、`apscheduler`（スケジューリング）、標準`sqlite3`、`anthropic`（AI）。
- データ取得（Step1〜3）：`yfinance`（遅延）、`jquants-api-client`（ヒストリカル）。LIVEは採用ブローカーのSDK（kabu＝REST/WS、IBKR＝`ib_insync`/`ib_async`）。
- 設定：`.env` ＋ YAML/TOML。

## 8. リスク管理・安全機構

> 完全自動売買で最も怖いのは相場で負けることだけでなく、**バグ・通信エラーによる誤発注**。
> 以下は最終関門（§5.4）として全注文に適用する。

### 8.1 損失・回数の上限
- 1日の最大損失額（到達で当日取引停止）。
- 1銘柄あたりの最大損失額。
- 1日の最大取引回数。
- 同一銘柄への連続エントリー制限・クールダウン。
- 1注文あたりの最大数量・最大想定金額。

### 8.2 発注・約定まわり
- **二重発注防止（冪等性）**：発注前に既存注文・ポジションを確認、発注キーで重複排除。
- 約定確認（発注しっぱなしにしない）。エントリー約定と同時に損切り逆指値を発注。
- 未約定注文の自動取消（タイムアウト）。
- 値幅制限・ストップ高/安時の挙動定義（成行回避・見送り）。
- 流動性・スプレッドチェック（板が薄い時は発注しない）。

### 8.3 接続・データ・障害
- APIエラー／WebSocket切断検知 → 新規発注停止＋通知、可能なら自動再接続。
- データ欠損・異常値（価格0、極端なスパイク）検知時はその銘柄の判定を一時停止。
- ポジション/残高の定期照合（§5.6）。
- 想定外例外は**フェイルセーフ＝停止**側に倒す。

### 8.4 時間・手動制御
- 大引け前の強制決済（持ち越さない）。
- 手動キルスイッチ（即時の新規発注停止）。
- 稼働モード初期値 `PAPER`、`LIVE` へは明示操作でのみ移行。

### 8.5 記録
- 全注文・全判断・全エラーのログ保存（事後検証・改善の前提）。

## 9. 取引制度・法規制上の前提（日本株）

> ⚠️ 本書は技術仕様であり、投資・税務・法務上の助言ではない。実運用前に証券会社の規約・最新制度を確認すること。

### 9.1 差金決済規制と口座方針
- 現物（特定/一般）口座では、**同一銘柄を同一日に同じ資金で何度も「売って買い直す」回転売買はできない**（差金決済の禁止）。
- 本システムは**現物口座**を用い、**同一銘柄は1日1往復まで**に制限する（§5.2）。トレード回数は**複数銘柄**に分散して確保する。
- 同一銘柄を1日に何度も回転させたい場合は信用取引が必要だが、レバレッジでリスクが増すため当面は採用しない（将来検討）。

### 9.2 取引時間
- 前場 9:00–11:30 ／ 後場 12:30–**15:30**（2024/11/5に延長）。引けに**クロージング・オークション**あり。スケジューラ（§5.10）はこれを前提とする。

### 9.3 値幅制限
- ストップ高/安、特別気配、値幅制限到達時は約定挙動が通常と異なる。成行発注のリスクを考慮する（§8.2）。

### 9.4 売買単位と銘柄選定 — ポジションを小さく保つ設計
- **自動・リアルタイムのデイトレは原則100株単位（単元）**で行う。
- **単元未満株（S株/ワン株/プチ株/かぶミニ等＝1株単位）は本システムには不向き**：約定タイミングが寄付など限られ、リアルタイムに任意価格で売買できず、API自動発注の対応も限定的（証券を変えても同様）。
- よって**1回の建玉を小さくする手段は「株価の低い銘柄を選ぶ」**こと（例：株価100円→100株=1万円、500円→5万円）。
- **値がさ株は避ける**：1株2万円超の銘柄は100株で200万円超となりMVPには過大。デイトレ人気の値がさ株は値動きの参考に留め、実弾は低位株とする。
- **銘柄スクリーニング基準（システムで自動選定）**：株価が低め（1単元が小さい）／ 出来高・売買代金が十分（流動性・狭いスプレッド）／ 値動き（ボラ）がそこそこある／ ストップ高安・極端な薄商いを除外。候補は機械的に抽出し、最終採否は人が確認する。

### 9.5 税・コスト
- 売買手数料、税金（特定口座源泉徴収等）を損益計算（`trades.fee`/`pnl`）に反映する。
- IBKRなど外国証券は特定口座が無く、確定申告が必要（§6.1）。

## 12. 開発ロードマップ

### Step 0：事前準備
- Mac開発環境構築（Python、`yfinance` 等）。VWAP順張りの簡易バックテスト（ヒストリカル分足はJ-Quants等で入手検討）。
- 銘柄スクリーニング基準（§9.4）で初期候補を抽出。

### Step 1：監視システム（Mac）
- データ取得・指標計算（VWAP/MA/出来高）・条件判定、**シグナル通知のみ**（実注文なし）。

### Step 2：仮想売買（フォワードテスト・Mac）
- `PaperBroker` で実発注せず損益を記録。Macのみ・本番証券不要。VWAP順張りのルールが機能するか確認・数値調整。

### Step 3：AI補助を追加（Mac）
- シグナル時にAIへ数値を渡し、Go/見送り/サイズを取得（タイムアウト/フォールバック付き）。全AI判断をログして当否を測る。最終判断はルールベース。

### Step 4：少額で実発注（LIVE）
- **WindowsデスクトップでkabuステーションAPIに接続**（まず検証環境で動作確認）。Macで作ったコードをgitで持ち込み、`PaperBroker`→`KabuBroker`へ差し替える。
- 1銘柄／100株／1日数回まで／最大損失制限あり。現物・低位株（§9.4）で開始。

### Step 5：改善・拡張
- 複数銘柄・複数戦略対応、日次レポート、勝率・損益分析、AIによる改善提案。
- 効果測定のうえ Level 3（AIが候補も提案）への拡張、VPS移行を検討。

```
Step0 準備 → Step1 監視 → Step2 仮想 → Step3 AI補助 → Step4 少額実弾 → Step5 拡張
（各Stepでリスク管理機構を先に入れてから次へ進む）
```

## 13. 日次運用フロー

1. 寄付前：自動起動・接続確認・当日リスク状態リセット・対象銘柄ロード。
2. 場中：監視 → シグナル → AI補助 → ガード → 発注/決済。各イベントをDB＋通知。
3. 大引け前：保有玉を強制手仕舞い。
4. 引け後：約定確定、日次集計（`daily_stats`）、AIによる日次分析・改善提案、サマリ通知。
5. 異常時：いつでもキルスイッチ／自動フェイルセーフ停止＋CRITICAL通知。
